### What Phase B's all_sum probe tells us

Per-layer all_sum CPU-wall in **steady-state warm decode** is uniform:
- Layers 0-42 (43 total): each fence-eval costs ~1.0-1.7ms (alternating
  by compress_ratio pattern)
- p99/p50 within 5% — tight distribution
- NO localized hotspot (the apparent 28-39 hotspot was iter-0
  cold-compile cache artifact)

So verify cost = 43 layers x ~1.4ms drain = **~60ms structural floor**.
At gamma=2 the cycle = 4.5ms draft + 60ms verify + 1ms accept = ~65ms.
With alpha=0.52 yielding 2.04 tokens/cycle, that's **~31 t/s
theoretical** without any further optimization. We measure 30.06 in
production -> 95% of the structural floor, suggesting the remaining
~5% gap (~1.5 t/s) is overhead.

**Implication: to hit 35 t/s, we need at least one of:**
- Reduce per-layer cost below 1.4ms (e.g., the ack_sync_pre + ACK QP
  fix may do this IF we can deploy it cleanly).
- Raise alpha (acceptance per cycle) — gamma=3 already tried and lost.
- Reduce layer count — model architecture change, off limits.
- Shrink the all_sum payload (per-rank work) — TP shape change, risky.
- Find the historical regression that took us from claimed 31.5 (May 17)
  to actual 29.5 (today) and revert it.

### Why this is hard

The structural floor at the per-layer all_sum cost (Thunderbolt 5 RDMA
round-trip ~1.4ms per layer) is the hard limit unless we change the
collective topology or payload. Software-level compile/fuse tricks
have all failed in this session. The remaining real gains are:
1. **In the historical commit history** — undoing accidentally-landed
   regressions.
2. **In the mlx C++ collective path** — the very thing whose May-18
   deploy attempt regressed.
3. **In the topk=160 -> topk=512 cost gap** — there's 2.3 t/s in the

## Phase C: TOPK=512 vs TOPK=160 cost-gap investigation (~2 hrs)

**Objective:** Identify the +2.3 t/s gap between TOPK=160 (32.35) and
TOPK=512 (30.06). Even if Phase B lands us at 32.3 at TOPK=160, we
still need to close this gap to hit 35 at TOPK=512.

### Hypothesis

The Indexer at line 1569 selects top-k positions; downstream
`_sparse_pooled_attention` gathers k items per query. Larger k =>
larger gather + larger sparse attention. The cost should scale roughly
linearly with k for the gather + matmul steps.

160 -> 512 is 3.2x. If sparse-pool attention is 100% bottlenecked on
gather, that's 3.2x the verify time -> would be catastrophic. We
observe verify going from ~52ms (TOPK=160 implied) to ~57ms (TOPK=512
measured) -> just 1.1x. So the linear-in-k cost is NOT the dominant
verify path. Something else is happening.

Alternative: the **fused topk Metal kernel** (`_fused_topk`) only
applies when `k <= 1024 AND scores.shape[1] == 1 AND pmask is None`.
At TOPK=160 with `EXO_DSV4_TOPK_FUSED=1` enabled, decode (L=1) uses the
fused kernel. At TOPK=512 same — both go through fused at decode. So
that's NOT the gap.

Verify (L=3 + pmask) bypasses fused topk regardless of k. So at verify
the only k-dependent cost is in:
1. The argsort: O(P * log P), independent of k.
2. The `[..., :k]` slice: O(P) regardless.
3. The downstream gather + sparse-attn matmul: O(k * D * H).

The matmul is O(k); at k=512 vs k=160 that's 3.2x for one matmul. But
the matmul is one of many ops per layer.

### Step 1: Read `_sparse_pooled_attention_inner` carefully

Find every place k appears in the layer compute. Estimate cost in
GFLOP/byte for each. The L_q>1 path uses the inner kernel which is
the manual split-softmax with 10+ ops.

### Step 2: Instrument with EXO_DSV4_TOPK env-var sweep

Quick A/B: run 2-iter benches at TOPK in {160, 256, 384, 512}. Plot
agg_tps vs k. If it's monotonic-decreasing, the cost is intrinsic to
k. If it's bimodal (jumps between k thresholds), there's a code path
gate we can find and tune.

### Step 3: Identify whether the TOPK=160 quality issue is real

The plan I'm following says "TOPK=160 quality unvalidated." Per the
quality-probe skill, we should run the canonical quality probe at
TOPK=160 vs TOPK=512 and quantify the actual difference. If quality
is acceptable at TOPK=256 or TOPK=384 (intermediate values), we might
have a usable quality/speed trade.


## Phase E: Architectural deep-dive (~5+ hrs, LAST RESORT)

**Only if Phases A-D combined don't reach 35.** This is where serious
engineering happens.

### E1: TP shape change

Currently shard hidden_dim. Try sharding heads (n_heads=64; rank
holds 32 heads each). This changes the all_sum payload size and may
hit different cost characteristics on Thunderbolt 5.

Rough math: currently each all_sum payload per layer is roughly
`B * L * hc_mult * hidden_size * dtype_size = 1 * 3 * 4 * 4096 * 2 = 100 KB`.
At Thunderbolt 5 ~80 Gb/s = ~10 GB/s = 10 us per layer just for the
data, plus the round-trip latency. Current 1.4ms per layer is dominated
by RTT, not bandwidth. Smaller payload won't help much.

If we shard heads instead, each rank holds 32 heads. The KV projection
output per rank is `B * L * hc_mult * 32 * head_dim`. Total all_sum is
the same... but the SHAPE of the workload per rank is different.

Probably not a big lift. Cost: 1-2 days to refactor + re-train weight
loaders + validate quality.

### E2: mlx C++ allreduce optimization

The current allreduce on UC QPs has known race issues (the
ack_sync_pre saga). If we can rewrite it to use RC instead of UC,
the race goes away and might be faster too. Cost: deep mlx internal
work, weeks.

### E3: Profile + GPU% diagnosis at the kernel level

Use Apple's xctrace / Metal System Trace to capture per-kernel GPU
utilization during decode. Identify any kernel that's at <50% GPU
utilization — that's where there's recoverable cycles.

### Phase E success criterion

We have actionable engineering plans, not actually hitting 35 t/s in
this session. This is multi-day work.

## Risks, Tradeoffs, Open Questions

### Risks

1. **Phase A bisect could be uninformative** if the May-17 31.5 was
   1-sample noise. Mitigation: time-box the bisect to 3 hours.

2. **Phase B's deploy hazard.** The May-18 12:48 regression hasn't
   been root-caused. Re-attempting the deploy has a real chance of
   wedging the cluster for 10+ min READY checks. Mitigation: small-
   prefill bench BEFORE 100K, abort on READY > 2 min.

3. **Phase C might find that TOPK=512 is structurally 2.3 t/s slower
   than 160 with no recoverable optimization.** Mitigation: in that
   case the only path to 35 is Phase B + (Phase E or accept-the-floor).

4. **Per-stream 35 t/s at c=1 might genuinely be impossible** at this
   hardware. The bandwidth floor is ~33 t/s (5.5ms/token at 400 GB/s
   for the 2.2 GB activation read). 35 t/s = 28.5ms/token. That's
   *5x* the bandwidth floor — should be possible but every dispatch
   us matters at that scale.

### Tradeoffs

- **Phase A vs Phase B order.** A is cheaper (bisect, no code changes)
  but lower individual EV (+1.5 t/s). B is higher risk but higher EV
  (+2.3 t/s). I'm starting with A because it informs B (e.g., if A
  finds the regression is in mlx wheel build, that IS the Phase B
  problem).

- **Phase D (concurrency) sidesteps the per-stream goal.** Worth doing
  for ~1 hr to see if it's a "good enough" answer. User may say "yes
  aggregate is fine" or "no, per-stream only."

### Open Questions

1. Is the goal per-stream 35 or aggregate 35? Phase D probes this.
2. Is the 30.06 -> 32.35 TOPK gap permanent or fixable? Phase C
   probes.
3. Did the May-18 02:02 32.29 number actually run on the fix-branch
   binary, or was it a measurement artifact? Phase B step 5 answers
   definitively.
