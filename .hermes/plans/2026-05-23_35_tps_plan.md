# Plan: c=1 100K MTP-on → 35 t/s (2026-05-23)

Current state: ~29.7 t/s c=1 100K MTP-on at production defaults (γ=2 K=0
FENCE=4 TOPK=512 bf16 KV MTP=1). Spec env-knob space is saturated.
Need +18% to hit 35.

## Hard constraints (user-enforced; do NOT cross)

- bf16 KV (no quantization)
- TOPK=512 (no reduction)
- Model defaults as shipped
- No mitigations — only root-cause fixes
- Quality validated at every step with c-aware probe + BOS detection

## Math

At 29.7 t/s with γ=2 K=0 accepting 1.07/2 drafts:
- Per-token wall: 33.7ms
- Per-spec-cycle yields: 1.07 + 1 bonus = 2.07 tokens
- Per-spec-cycle wall: 33.7 × 2.07 ≈ 69.7ms
- Per May-12 V4Block profile breakdown: verify ≈ 93% of cycle = ~65ms,
  draft chain ≈ 5.5% = ~3.8ms, accept/commit ≈ 1.5%

Two independent paths to 35 t/s:

**Path A — Reduce verify wall by 15%.** 65ms → 55ms verify → ~28.5ms/token → 35 t/s.
**Path B — Raise MTP acceptance by 30%.** 1.07/2 → 1.4/2 → 2.4 tokens/cycle → 34.5 t/s.

Either path alone clears the target. They stack if both hit; +5% from
Path A combined with +20% from Path B would land at 37+ t/s with margin.

## Workstreams (ordered by expected reward / hour)

### W1 — Fix the c=2 100K spec-batched bug (HIGHEST PRIORITY)

**Why this first**: It's a real production bug, well-localized, and
fixing it makes the c=2 aggregate path work — which is what the
cluster was originally sized for. **c=2 at ~17 t/s/stream aggregates
to ~34 t/s** at production defaults if both streams run clean, and
the existing c=1 wins compound naturally. This is the cheapest path
to the aggregate goal even if c=1 single-stream stays at 29.7.

**Scope**:
1. Add per-step per-stream instrumentation INSIDE `_speculative_next_batch`
   (not just `_draft_tokens_batched` — the tracer at `a1caaeb3` only
   covers draft, not verify+accept). Log per-stream: `n_accepted_per`,
   `target_tokens`, `draft_concat`, `verify_logits` first-row argmax,
   `bonus_vals`. Find which produces garbage on iter 1 of c=2 100K.
2. Add the same per-stream eval inside `dsv4_speculative_forward` —
   inspect `verify_logits[stream_b, :, :]` for both streams before
   acceptance. If one stream's logits are already garbage in the
   verify forward output, the bug is upstream (cache state, model
   forward at long context with per-stream cache). If verify logits
   are clean and acceptance produces garbage, the bug is in
   `_speculative_next_batch`.
3. Test the `_upgrade_cache_to_per_stream` swap timing: dump cache
   state immediately before and after the swap; verify per-stream
   `offset` and `_per_stream_max` align with what `_update_concat`
   left.

**Time estimate**: 4-12 hours of code-read + 2-3 cluster cycles
(15 min each).

**Quality gate**: c=2 100K with `--concurrency 2 --iters 3` on the
v2 probe (`fe837b49`). Both streams: needle ✓, 0 BOS, finish=stop.

**Expected outcome**: If c=2 100K decode works, aggregate hits ~32-34
t/s naturally (2× the validated c=2 per-stream we measured 2026-05-12
post-profiler-removal: 16.2 t/s/stream). Stack fused-topk and we're
at 33-35.

### W2 — Validate fused-topk properly at higher iter count (CHEAP +1 t/s)

**Why**: Today's 3-iter measurement was -3% vs baseline within ±0.7
noise. The microbench claims 5× speedup per call = ~3% expected lift
at decode (~1ms/token saved over 43 layers × ~half being sparse).
That's ~0.9 t/s lift, sitting below today's measurement floor.

**Scope**:
1. Run 10-iter c=1 100K bench at γ=2 K=0 WITHOUT fused-topk —
   establish σ baseline
2. Run 10-iter c=1 100K bench at γ=2 K=0 WITH `EXO_DSV4_TOPK_FUSED=1`
3. Compute σ-difference; if >2σ separation, real win
4. If real: add `EXO_DSV4_TOPK_FUSED:=1` to `start_cluster.sh` defaults
5. If marginal: document the limit and move on

**Time estimate**: 1.5 hours (two 30-min benches + analysis).

**Quality gate**: 100K needle probe ✓, 0 BOS on every iter.

**Expected outcome**: +0.5-1.0 t/s validated; default flip if real.

### W3 — Eagle implementation audit (acceptance ceiling)

**Why K=8 didn't lift acceptance is suspicious.** Theory: top-K soft-mix
should help when top-1 isn't dominant. Two possibilities:

a) DSv4's MTP head IS so peaked that top-1 dominates the softmax → Eagle is
   a structural no-op for this model
b) Eagle implementation has a subtle bug nullifying the lift

If (a), Eagle is dead — move on. If (b), fixing it could push acceptance
into 1.3-1.4/2 territory.

**Scope**:
1. Dump MTP top-K logit distribution at one decode step at 100K. If
   top-1 probability > 0.9 consistently, Eagle is dead. If 0.6-0.8,
   there's room.
2. Verify Eagle's soft_emb assembly matches the math: `soft_emb =
   sum(softmax(top_K) * embed(top_K))`. Diff against the K=1 fast-path
   (which collapses to `embed(argmax)`).
3. Verify `prev_logits` capture timing — if the chained predict captures
   logits BEFORE Eagle's broadcast point, Eagle is using stale logits.
   (This was the May-22 bug fixed by commit `40b6e9c0`; verify it's
   really fixed in the current code path.)
4. If implementation is correct AND distribution is peaked: Eagle is
   structurally dead, document and move on.

**Time estimate**: 4-6 hours.

**Quality gate**: short-prompt smoke + 100K needle probe at K=4 vs K=8
vs K=0 to compare acceptance histograms.

**Expected outcome**: Either +5-15% throughput from Eagle working, or a
definitive "Eagle is dead for this model" entry in the don't-re-attempt
list (saving future time).

### W4 — Verify-forward kernel reduction (the structural lever)

**Why**: 65ms verify at 100K is most of the wall. Per May-12 V4Block
probe, attention is 96% of per-layer wall (sparse-indexer O(L) score
pass). 15% reduction = clears 35 standalone.

**Scope**:
1. Re-run V4Block per-section probe (`EXO_PROFILER=spans`) at TODAY's
   defaults to refresh the breakdown. Per pitfall #17 the probe itself
   costs ~40%, but the RATIO data is real even at slow wall.
2. Identify the top time-consumer at layer level. Likely candidates:
   - `_indexer_score` — the O(L_pool) dot product
   - `_sparse_pooled_attention` — SDPA over selected top-K positions
   - `_rope_dispatch` — per-layer RoPE application
   - The per-layer `all_sum` + `mx.eval` fence pair
3. Look for:
   - Redundant casts (float ↔ bf16) — pitfall #16 says compile on
     already-compiled primitives hurts; bf16 indexer score was already
     a +3.4% win (May 11)
   - Missing `mx.compile` boundaries where they'd help
   - Pre-computable values being recomputed per layer
   - Per-layer `all_sum` overlap with next layer's pre-attn compute

**Time estimate**: 8-16 hours. This is the slowest-burn workstream
but also the highest-ceiling — past wins at this layer (`5518f17`,
`ee68e44`, `1a0aa5c`, `87f4625` for sparse SDPA matmul) have produced
10-20% one-shot wins.

**Quality gate**: 100K needle probe ✓, 0 BOS, finish=stop. Compare
decode rate at fixed σ < 0.3 over 10 iters.

**Expected outcome**: +5-15% if a real lever is found. Compounds with
W2 fused-topk.

### W5 — All_sum overlap with next layer's compute (LATENT lever)

**Why**: Each layer does `all_sum → mx.eval` then the NEXT layer starts.
There's potentially overlap room: if we could begin the next layer's
HC + RMSNorm + Q/K/V projection while the all_sum is still
draining, we hide the comm wall.

**Scope**:
1. Read `DeepseekV4Block.__call__` carefully to see what's truly
   data-dependent on the prior layer's all_sum output.
2. The HC + RMSNorm + Q proj all read y. So strictly speaking everything
   depends. BUT: if the model uses pre-normalization, the input to
   the next layer IS y_prev directly, and we'd need a `mx.depends_on`-
   shaped split.
3. If feasible, prototype on one layer pair and measure.

**Time estimate**: 12+ hours. Very speculative — mlx's lazy graph
may already do this implicitly. The `mx.eval(y)` fence at line 1317-1335
of `DeepseekV4MoE.__call__` is what FORCES the wait. Removing or
relaxing it without breaking c=2 lockstep is the question.

**Quality gate**: same as W4.

**Expected outcome**: Unknown. Could be the biggest win or 0. Save
for last.

## Recommended sequence

1. **W2 first (1.5h)** — quick measurable win, validates the testing
   protocol at higher iter count.
2. **W1 in parallel** — c=2 fix is the highest-EV workstream. If it
   works, it unlocks aggregate target without needing W3-W5.
3. **W3 (4-6h)** — Eagle audit. Either delivers acceptance lift OR
   definitively kills Eagle as a future lever. Either way it's
   high-information.
4. **W4 (8-16h)** — kernel-level work, the actual structural lever.
   Only pursue this if W1+W2+W3 didn't land us at 35 yet.
5. **W5 (12+h)** — speculative; only if W1-W4 fall short.

## Stop-light criteria

After W1+W2+W3 (15-20 hours work), reassess:
- If c=2 100K works + W2/W3 deliver: aggregate target hit, done.
- If c=2 100K still broken AND W3 says Eagle is dead AND W2 is +1 t/s:
  need W4. Time-box W4 to 2 days; if no structural win surfaces,
  recommend the user lower the 35 target or accept the c=1 ceiling
  while continuing c=2 work in background.
